# 当前主线方案完整说明（方案三）

## 1. 当前主线结论

当前最推荐继续推进的路线，不再是：

- 纯方案二：`wav2vec + 原始 full-rank 第二阶段`

而是：

- **方案三 = 方案一 + 方案二联合**
- **第一阶段：wav2vec 音频编码 + diffusion backbone**
- **第二阶段：low-rank PWSG 个性化权重编辑**

一句话总结：

- 第一阶段保留方案二的优势，用更强的音频表示替代原始 MFCC
- 第二阶段不再使用原始论文的 full-rank 权重编辑器，而改为方案一中的 low-rank PWSG
- 这样既保留 wav2vec 的信息增益，也规避第二阶段 full-rank 的高显存和高计算开销

当前主线的判断是：

- `wav2vec` 本身是成立的
- `wav2vec + full-rank 第二阶段` 不适合当前硬件
- 主实验应固定为：`wav2vec + low-rank PWSG`

---

## 2. 方案关系

### 2.1 方案一

方案一的核心，是把第二阶段的个性化权重编辑器从 full-rank 改成 low-rank。

原始方式直接预测完整的 `ΔW`，开销很大。  
方案一改成预测低秩因子：

- `A ∈ R^(d_out × r)`
- `B ∈ R^(r × d_in)`
- `ΔW = (alpha / rank) * A @ B`

这样显著降低第二阶段的参数量、显存开销和训练负担。

### 2.2 方案二

方案二的核心，是把第一阶段的音频输入从 MFCC 改成 `wav2vec2`。

也就是说：

- 第一阶段不再使用手工特征 MFCC
- 改为使用 `Wav2Vec2AudioEncoder`
- 让 diffusion backbone 学到更强的音频条件表示

### 2.3 方案三

方案三不是一个全新独立体系，而是把前两者的优点拼起来：

- 第一阶段用方案二：`wav2vec`
- 第二阶段用方案一：`low-rank PWSG`

所以方案三的真实含义是：

- **第一阶段产物 = wav2vec 版 diffusion backbone**
- **第二阶段产物 = 在该 backbone 上训练出来的 low-rank 个性化权重编辑器**

## 3. 当前代码里已经落地的实现

当前仓库已经把方案三所需的关键实现补齐，涉及以下文件。

### 3.1 音频编码器

文件：

- `model/audio_model/wav2vec.py`

实现内容：

- 新增 `Wav2Vec2AudioEncoder`
- 基于 HuggingFace `Wav2Vec2Model` 初始化
- 输入是 `[batch, frames, samples_per_frame]`
- 内部会先把逐帧波形拼成连续音频，再送入 wav2vec
- 通过线性插值把特征对齐到视频帧率 `25 fps`
- 最后通过一个线性层投影到 `output_dim=256`

当前实现细节：

- `freeze_feature_encoder=True`
- `freeze_backbone=True`
- 第一阶段 `train_projection=True`
- 第二阶段 `train_projection=False`
- `apply_layer_norm=True`

这意味着：

- 第一阶段不是整套 wav2vec 全量微调
- 实际上主要训练的是后接的投影层，以及 diffusion 主体
- 第二阶段直接固定 wav2vec 编码器参数

### 3.2 数据集支持 waveform

文件：

- `dataset/dataset.py`

当前实现支持两种音频表示：

- `audio_representation: features`，走原始 MFCC 路线
- `audio_representation: waveform`，走 wav2vec 路线

当使用 `waveform` 时，处理流程是：

1. 用 `torchaudio.load` 读取音频
2. 如果是双声道，先转单声道
3. 如果采样率不是 `16000`，重采样到 `16000`
4. 对波形做零均值化
5. 按 `fps=25` 切成逐帧音频片段
6. 如果长度不足则补零，过长则裁剪
7. 输出形状为 `[n_frames, samples_per_frame]`

这一步是方案三的基础，因为 wav2vec 输入不再是 MFCC，而是对齐后的一段段 waveform。

### 3.3 backbone 训练链路修正

文件：

- `model/diffusion/matchers.py`

关键点：

- 新增 `encode_audio`
- 如果 `audio_encoder` 有可训练参数，训练时不再被 `torch.no_grad()` 截断
- prior 和 decoder 共享同一份 `audio_encoder`

此外还新增了：

- `maybe_load_pretrained_checkpoint`

逻辑是：

- `mode == "test"` 时加载 checkpoint
- 或者配置里 `load_pretrained_backbone: True` 时也加载 checkpoint

这对第二阶段非常关键。  
原因是第二阶段训练时，必须显式加载第一阶段已经训练好的 wav2vec backbone，否则就会拿随机初始化的 prior/decoder 去做个性化训练，实验不成立。

### 3.4 第一阶段 checkpoint 保存修正

文件：

- `utils/util.py`

当前 `save_checkpoint` 会保存三部分：

- `DiffusionPriorNetwork/checkpoint.pth`
- `TransformerDenoiser/checkpoint.pth`
- `Wav2Vec2AudioEncoder/checkpoint.pth`

这一步非常关键，因为第二阶段不只需要 prior 和 decoder，还需要与之配套的 wav2vec 编码器。

### 3.5 第二阶段 low-rank PWSG

文件：

- `model/modifier/network.py`

当前实现包括：

- 原始 `ModifierNetwork`
- 新增 `LowRankModifierNetwork`
- `MainNetUnified` 支持通过配置切换 full-rank 或 low-rank

low-rank 实现方式是：

- 共享若干层 MLP 作为超网络前端
- 对每个目标层分别预测 `A` 和 `B`
- 使用 `scale = alpha / rank`
- 最终重构 `ΔW = scale * (A @ B)`

当前实现细节：

- 只支持 `predict: shift`
- 第二阶段默认 `rank=8`
- 第二阶段默认 `lora_alpha=8.0`
- `B` 分支参数初始化为 0
- 当前超网络要求 `batch size = 1`

`MainNetUnified` 的工作方式是：

1. 用 `person_encoder` 从个性先验 `p` 里提取个性向量
2. 用 hypernet 生成目标层的 `ΔW`
3. 动态替换目标模块的 `weight` 或 `in_proj_weight`
4. 再跑主干 diffusion 网络

当前默认修改的层是：

- `diffusion_decoder.model.decoder.layers.4.multihead_attn`
- `diffusion_decoder.model.decoder.layers.6.multihead_attn`
- `diffusion_decoder.model.to_emotion_feat`

也就是说，方案三第二阶段并不是重训整个网络，而是在固定 backbone 的基础上，给这些层动态注入个性化低秩权重偏移。

### 3.6 第二阶段训练脚本修正

文件：

- `train_rewrite_weight.py`

当前逻辑是：

- 实例化第一阶段 diffusion 主干
- 再包装成 `MainNetUnified`
- 只构建并优化 `main_model.hypernet.parameters()`
- 每个 iteration 调用 `optimizer_hypernet.zero_grad(set_to_none=True)`
- 反向传播后只执行 `optimizer_hypernet.step()`

这意味着：

- 第二阶段只训练 low-rank hypernet
- 不再更新主干 network
- 这和方案三的设计目标一致

### 3.7 测试脚本

文件：

- `evaluate_rewrite_weight_all.py`

当前支持：

- `--protocol mafrg`
- `--protocol pmafrg`
- `--protocol both`
- `--compute_frrea`
- `--frrea_period`
- `--frrea_step`
- `--frrea_fake_repeat`

评测脚本会：

1. 加载第一阶段 backbone
2. 加载第二阶段 `LowRankModifierNetwork`
3. 对每个样本生成 10 次预测
4. 计算 FRCorr、FRdist、FRDvs、FRVar、FRDiv、FRSyn
5. 如果开启 `--compute_frrea`，还会渲染视频并计算 `FRRea(FID)`

## 5. 第二阶段：wav2vec + low-rank PWSG

### 5.1 第二阶段要解决什么问题

第二阶段解决的是：

- 原始第二阶段 full-rank PWSG 过重
- 显存占用高
- 反向传播成本大
- 在当前硬件上容易 OOM

方案三第二阶段改成：

- 固定第一阶段 backbone
- 只训练 low-rank 权重编辑器

### 5.2 第二阶段的核心公式

原始 full-rank 方式是直接预测完整权重偏移：

- `ΔW`

low-rank 方式改成：

- 预测 `A`
- 预测 `B`
- 重构 `ΔW = (alpha / rank) * A @ B`

默认超参数：

- `rank = 8`
- `lora_alpha = 8.0`

```bash
nohup python evaluate_rewrite_weight_all.py --mode test --config configs/rewrite_weight_wav2vec.yaml --protocol both --compute_frrea --frrea_period 10 --frrea_step 1 --frrea_fake_repeat 10 --frrea_dir ./results/rewrite_weight_wav2vec/frrea_eval > result_wav2vec.log 2>&1 &
```

查看日志：

```bash
tail -f result_wav2vec.log
```

### 6.4 为什么完整测试里建议显式传 `--frrea_dir`

这是当前脚本里的一个细节。

虽然第二阶段配置里有：

- `trainer.out_dir: ./results/rewrite_weight_wav2vec`

但 `evaluate_rewrite_weight_all.py` 的 `FRRea` 输出目录并不是从 yaml 自动读取的，脚本默认值实际上是：

- `./results/rewrite_weight/frrea_eval`

因此如果你不显式传：

- `--frrea_dir ./results/rewrite_weight_wav2vec/frrea_eval`

那么完整测试的渲染结果和 FID 目录可能会落到默认的 `rewrite_weight` 目录，而不是 `rewrite_weight_wav2vec` 目录。

所以当前主线里建议始终显式带上 `--frrea_dir`。

### 6.5 `--protocol` 的含义

支持三种协议：

- `mafrg`
- `pmafrg`
- `both`

含义可以理解为：

- `mafrg`：通用测试协议
- `pmafrg`：person-specific 测试协议
- `both`：两者都跑

如果你要做正式汇报，建议直接用：

- `--protocol both`

### 6.6 `FRRea` 相关参数

#### `--compute_frrea`

开启后：

- 会触发渲染
- 会调用 `compute_FRRea`
- 耗时显著增加

#### `--frrea_period`

- 每隔多少个 batch 取一次样本做 FRRea 渲染

#### `--frrea_step`

- 渲染时的帧步长

#### `--frrea_fake_repeat`

- 每个选中 batch 最多取多少个 fake sample 用于 FRRea 渲染

当前推荐值：

- `frrea_period = 10`
- `frrea_step = 1`
- `frrea_fake_repeat = 10`

### 6.7 测试脚本里一个容易忽略的细节

虽然第二阶段 `test_dataset.k_appro = 1`，但测试脚本内部仍然会：

- 对每个样本生成 10 次预测

这是因为脚本内部显式循环了 10 次采样，用来计算多样性相关指标和 FRRea。

因此测试时最终统计的不是单次预测，而是 10 次采样的集合结果。

### 6.8 测试产物

快速测试通常不会写太多文件，主要是：

- 终端输出
- 如果用了 `nohup`，则输出到 `result_wav2vec.log`

完整测试通常会产生：

- `result_wav2vec.log`
- `./results/rewrite_weight_wav2vec/frrea_eval/...`

其中 `frrea_eval` 下通常会有：

- 渲染图片
- `fid` 目录
- 用于 `FRRea(FID)` 计算的中间文件

## 9. 当前方案中的关键注意事项

### 9.1 第一阶段产物不是 yaml

第一阶段真正的训练产物是 checkpoint，不是 yaml。  
yaml 只是配置，不是模型结果。

### 9.2 第二阶段训练产物也不是 yaml

第二阶段真正的训练产物是：

- `checkpoints/rewrite_weight_wav2vec/LowRankModifierNetwork/checkpoint.pth`

测试时用的还是 `yaml`，但 `yaml` 的作用是告诉脚本去哪里加载这些 checkpoint。

### 9.3 远程服务器路径不一致时必须改配置

如果远程服务器上的第一阶段三个 `pth` 不在：

- `./checkpoints/diffusion_model_wav2vec/...`

而在别的目录，那么至少要同步修改：

- `pretrained_checkpoint_dir`
- `audio_encoder.checkpoint_path`

### 9.4 当前实现里 `out_dir` 不是训练主产物目录

两个训练脚本当前真正直接写出的核心内容是：

- checkpoint
- log
- tensorboard

`trainer.out_dir` 并不是训练脚本当前最核心的落盘位置。  
真正会主动往结果目录写较多内容的，主要是开启 `FRRea` 的评测流程。

### 9.5 第二阶段只适合 batch size 1

当前 `LowRankModifierNetwork` 和 `ModifierNetwork` 都要求：

- `batch size = 1`

因此第二阶段配置里：

- `dataset.batch_size: 1`

这是当前实现约束，不建议随意改大。

### 9.6 当前第二阶段 low-rank 只支持 `predict: shift`

当前代码里有显式限制：

- 如果 `use_low_rank=True`
- 那么 `predict` 只能是 `shift`

因此不要把第二阶段改成：

- `predict: offset`
- 或 `predict: weight`

除非同步改代码。

### 9.7 第二阶段保存是覆盖式的

当前第二阶段每到保存周期，都会写到同一路径：

- `checkpoints/rewrite_weight_wav2vec/LowRankModifierNetwork/checkpoint.pth`

因此：

- 它不是多版本快照目录
- 而是最新一次保存覆盖旧 checkpoint

如果你需要保留多个 epoch 版本，建议额外手动备份。

## 11. 最后总结

当前这条主线的正确理解是：

- 第一阶段训练 `wav2vec` 版 diffusion backbone
- 第一阶段产物是三个 checkpoint
- 第二阶段在这三个 checkpoint 基础上继续训练 low-rank 个性化权重编辑器
- 第二阶段产物是 `LowRankModifierNetwork/checkpoint.pth`
- 测试阶段通过 `configs/rewrite_weight_wav2vec.yaml` 同时加载第一阶段 backbone 和第二阶段 hypernet

因此，当前方案三的全流程可以概括为：

1. 准备 wav2vec 配置和本地预训练模型
2. 训练第一阶段，得到三个 wav2vec backbone checkpoint
3. 用第二阶段配置加载这三个 checkpoint
4. 训练 low-rank PWSG，得到 `LowRankModifierNetwork/checkpoint.pth`
5. 先做快速测试，再做完整测试
6. 保存 checkpoint、日志、FRRea 结果和最终指标

当前最推荐继续推进的主线，仍然是：

- **第一阶段用 wav2vec**
- **第二阶段用 low-rank PWSG**
