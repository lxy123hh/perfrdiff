# 当前主线方案总结（方案三）

## 1. 当前结论

当前最推荐继续推进的路线，不再是：

- 纯方案二：`wav2vec + 原始 full-rank 第二阶段`

而是：

- **方案三 = 方案一 + 方案二联合**
- **第一阶段：wav2vec 音频编码**
- **第二阶段：low-rank PWSG**

这条路线的含义是：

1. 第一阶段保留方案二的优势，用更强的音频表征替代原始 MFCC。
2. 第二阶段不再使用原论文的 full-rank 权重编辑器，而改为方案一里的 low-rank PWSG。
3. 这样既保留了 wav2vec 的信息增益，也避开了第二阶段 full-rank 的大显存开销。

一句话总结：

- **方案二第一阶段成立**
- **方案二配原始 full-rank 第二阶段不适合当前硬件**
- **主实验应转为：wav2vec + low-rank PWSG**

---

## 2. 第一阶段产物和方案二的关系

是一样的。

更准确地说：

- 当前方案三中的第一阶段，本质上就是方案二的第一阶段。
- 也就是说，第一阶段训练完成后，产物仍然是 wav2vec 版本的 diffusion backbone。
- 因此，**方案三的第一阶段产物 = 方案二第一阶段产物**。

第一阶段跑完后，至少应有这三个 checkpoint：

- `checkpoints/diffusion_model_wav2vec/DiffusionPriorNetwork/checkpoint.pth`
- `checkpoints/diffusion_model_wav2vec/TransformerDenoiser/checkpoint.pth`
- `checkpoints/diffusion_model_wav2vec/Wav2Vec2AudioEncoder/checkpoint.pth`

其中前两个是 backbone，第三个是 wav2vec 音频编码器。

第二阶段训练时会在这三个产物基础上继续做 low-rank 个性化训练。

## 3. 当前仓库已经落到代码里的内容

当前仓库已经补齐了方案三所需的关键实现。

### 3.1 wav2vec 编码器

- `model/audio_model/wav2vec.py`
  - 新增了 `Wav2Vec2AudioEncoder`
  - 支持直接通过配置实例化
  - 支持输出 `output_dim=256` 的音频特征

### 3.2 数据集支持 waveform

- `dataset/dataset.py`
  - 保留原始 MFCC 路径
  - 新增 waveform 读取
  - 新增重采样、裁剪、对齐到视频帧的逻辑
  - 当配置里写：
    - `audio_representation: waveform`
    - `audio_sample_rate: 16000`
    时，会输出 wav2vec 可直接使用的音频张量

### 3.3 训练链路修正

- `model/diffusion/matchers.py`
  - 新增 `encode_audio`
  - 如果 audio encoder 有可训练参数，则训练时不再被 `torch.no_grad()` 截断
  - prior 和 decoder 共享同一份 audio encoder，避免第一阶段保存出两套不一致的音频编码器

### 3.4 checkpoint 保存修正

- `utils/util.py`
  - 第一阶段保存 checkpoint 时，除了：
    - `DiffusionPriorNetwork`
    - `TransformerDenoiser`
  - 还会额外保存：
    - `Wav2Vec2AudioEncoder`

这一步很关键，因为如果没有第三个文件，第二阶段就无法完整加载 wav2vec 音频编码器。

## 4. 整体思路

方案三的核心思想是：

### 4.1 第一阶段解决“输入表征不够强”的问题

原始 PerFRDiff 使用 MFCC 音频特征。

方案二的改进点是：

- 用 `wav2vec2` 替换原始 MFCC
- 让第一阶段 backbone 学到更强的音频条件表示

因此第一阶段的目标是：

- 训练一个 `wav2vec` 版本的 diffusion backbone
- 产物是：
  - `DiffusionPriorNetwork`
  - `TransformerDenoiser`
  - `Wav2Vec2AudioEncoder`

### 4.2 第二阶段解决“个性化权重编辑太重”的问题

原始第二阶段使用 full-rank PWSG。

问题是：

- full-rank 权重编辑矩阵太大
- 第二阶段反向传播时显存开销很高
- 在当前硬件上容易 OOM

方案一的改进点是：

- 把第二阶段的权重编辑器改成 low-rank 版本
- 即不直接预测完整 `ΔW`
- 而是预测低秩因子 `A` 和 `B`
- 再构造：
  - `ΔW = (alpha / rank) * A @ B`

因此第二阶段的目标是：

- 固定第一阶段已经训好的 wav2vec backbone
- 只训练 low-rank 个性化权重编辑器
- 生成：
  - `LowRankModifierNetwork/checkpoint.pth`

## 7. 关键细节

### 7.1 第一阶段和第二阶段的关系

第一阶段负责训 backbone。

第二阶段不重新训练整个 diffusion 模型，而是在第一阶段 backbone 基础上，做个性化权重编辑。

因此：

- 如果第一阶段产物已经完整，就不需要重复跑第一阶段。
- 第二阶段可以直接开始。

### 7.2 为什么第二阶段必须是 low-rank

因为已经验证过：

- `wav2vec + full-rank 第二阶段`
- 在当前硬件上容易 OOM

所以第二阶段必须改成：

- `low-rank PWSG`

### 7.3 为什么要共享 audio encoder

如果 prior 和 decoder 各自持有一份独立的 audio encoder：

- 第一阶段训练时可能会出现两套音频编码器权重
- 保存 checkpoint 时不容易统一
- 第二阶段加载时也容易不一致

现在代码已经改成：

- prior 和 decoder 共享同一份 `audio_encoder`

这样更符合“一个 wav2vec 编码器服务整个 backbone”的思路。

### 7.4 为什么要单独保存 `Wav2Vec2AudioEncoder`

因为第二阶段不仅需要 prior 和 decoder，还需要与之配套的 wav2vec encoder。

如果只保存：

- `DiffusionPriorNetwork`
- `TransformerDenoiser`

而不保存：

- `Wav2Vec2AudioEncoder`

那么第二阶段虽然能加载一部分 backbone，但音频编码链路并不完整。

## 12. 最后总结

当前主线方案已经很明确：

- 不再走“纯方案二 + full-rank 第二阶段”
- 继续推进“方案三 = wav2vec + low-rank PWSG”

第一阶段负责提升音频条件表示。

这条路线既保留了方案二的有效部分，也用方案一修复了方案二在第二阶段的训练瓶颈，是当前最合理、最稳妥、也最值得继续推进的主实验路线。
